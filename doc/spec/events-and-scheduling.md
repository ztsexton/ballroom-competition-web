# Events and Schedule Generation

This document describes the event model, registration flow, round generation, section events, and the schedule generation pipeline.

---

## 1. Event Model

An **Event** represents a single competitive category. Key fields:

| Field | Type | Description |
|-------|------|-------------|
| `id` | number | Auto-generated identifier |
| `name` | string | Display name (e.g., "Pro-Am Syllabus Bronze Smooth Waltz") |
| `designation` | string? | "Pro-Am", "Am-Am", etc. |
| `syllabusType` | string? | "Syllabus", "Open" |
| `level` | string? | "Bronze", "Silver", "Gold", etc. |
| `style` | string? | "Smooth", "Rhythm", "Standard", "Latin" |
| `dances` | string[]? | ["Waltz"], ["Waltz", "Tango", "Foxtrot"] |
| `scoringType` | string? | "standard" (ranking) or "proficiency" (numeric) |
| `isScholarship` | boolean? | Multi-dance championship event |
| `ageCategory` | string? | Age division |
| `heats` | Heat[] | Round structure with bibs and judges |
| `sectionGroupId` | string? | Links section events (A, B, C) together |
| `sectionLetter` | string? | Section identifier: "A", "B", "C" |
| `scratchedBibs` | number[]? | Couples withdrawn day-of |

**Two events are considered the "same type"** when all of these match: designation, syllabusType, level, style, dances (sorted), scoringType, and ageCategory.

## 2. Event Lifecycle

### Creation

Events can be created in two ways:

1. **Manual creation** via `POST /api/events` — admin specifies name, bibs, judges, and metadata
2. **Auto-creation** via `POST /api/events/register` — the registration service finds or creates an event matching the given combination

### Registration Flow (find-or-create)

When registering a couple for a combination:

1. Search all events in the competition for a matching type
2. If found and couple not already in it: add couple, rebuild heats
3. If not found: auto-create event with generated name from combination attributes
4. If `allowDuplicateEntries` is on: see Section Events below

### Couple Management

- **Add entry**: `POST /api/events/:id/entries` — adds bib to first-round bibs, rebuilds heats
- **Remove entry**: `DELETE /api/events/:id/entries/:bib` — removes bib, rebuilds heats
- **Late entry**: `POST /api/events/:id/late-entry` — adds to event even if scores exist
- **Scratch**: `POST /api/events/:id/scratch` — marks couple as withdrawn without removing them

## 3. Round Generation

Round structure is determined by couple count at event creation:

| Couples | Rounds |
|---------|--------|
| 1-6 | Final only |
| 7-14 | Semi-final, Final |
| 15+ | Quarter-final, Semi-final, Final |

**Proficiency events** always have a single Final round regardless of couple count.

When couples are added or removed, heats are rebuilt via `rebuildHeats()` which recalculates the round structure.

## 4. Section Events

### Purpose

In studio/pro-am competitions, a professional dancer often partners with multiple students at the same level. Both couples (Pro+Student1 and Pro+Student2) want to compete in the same event type, but the pro can't dance simultaneously in two heats.

### `allowDuplicateEntries` Setting

A competition-level boolean flag (`Competition.allowDuplicateEntries`). When enabled:

1. The registration service checks for **person conflicts** before adding a couple to an event
2. A person conflict exists when any person (leader or follower) in the incoming couple is already in the event through another couple
3. If conflict exists, the system creates a new **section event**

### Section Creation Flow

1. First couple registers normally (no sections yet)
2. Second couple with shared person triggers section creation:
   - Original event is renamed with " - A" suffix
   - Original event gets `sectionGroupId` and `sectionLetter: 'A'`
   - New event is created with " - B" suffix, same `sectionGroupId`, `sectionLetter: 'B'`
3. Third couple with shared person creates " - C" section, etc.
4. If a couple has no person conflict with any existing section, it joins the first conflict-free section

### Section Group ID

Format: `sg-<timestamp>-<random>`. Links all sections of the same event type together. Used by the schedule generator to prevent section events from being merged into the same scheduled heat.

### Results Behavior

Section events are independent events with their own scoring and results. Each section has its own final placement.

## 5. Schedule Generation Pipeline

The schedule is generated by `generateSchedule()` in `scheduleGenerator.ts`. The pipeline:

```
1. Sort events by style → type → level → dance
2. Build entry list (buildStyleBlockEntries or buildRoundDepthEntries)
3. Merge compatible entries into multi-entry heats (mergeEntries)
4. Split heats exceeding floor capacity (applyFloorHeatSplitting)
5. Insert auto-breaks between style changes (optional)
6. Separate adjacent heats sharing an event (separateEventRounds)
7. Minimize person-level back-to-back conflicts (minimizePersonBackToBack)
8. Auto-assign judges
9. Calculate timing estimates
```

### Step 1-2: Entry Building

**Default mode** (`deferFinals=false`): Groups events by style, then by type (single < multi-dance < scholarship). Within each group, entries are sorted by level then by round depth (quarter-final before semi-final before final).

**Deferred finals mode** (`deferFinals=true`): All first-round heats first, then all second-round, then all third-round.

### Step 3: Heat Merging (`mergeEntries`)

Uses **first-fit-decreasing bin packing**. Two entries can share a scheduled heat when:

- Same style, dances (sorted), and scoring type
- Combined couple count doesn't exceed `maxCouplesPerHeat`
- Neither event has multiple rounds (multi-round events are never combined)
- **Section events with the same `sectionGroupId` are never merged together**

### Step 4: Floor Heat Splitting

When `maxCouplesOnFloor` (or `maxCouplesOnFloorByLevel`) is set and a round has more couples than the floor capacity:

- Bibs are split into groups using `splitBibsEvenly()`
- Each group becomes a separate floor heat with `floorHeatIndex` and `totalFloorHeats`
- For multi-dance events: creates N groups x D dances heats, ordered by dance then group
- Floor heats are labeled with letters: A, B, C (not numbers)

### Step 5: Auto-Breaks

When `autoBreaks.enabled` is true, a break heat is inserted between style changes.

### Step 6: Separate Event Rounds (`separateEventRounds`)

Post-processing pass that looks for adjacent heats sharing an event ID and swaps with a nearby heat (2-5 positions ahead) to separate them, without creating new event adjacency.

### Step 7: Minimize Back-to-Back (`minimizePersonBackToBack`)

Greedy adjacent-swap optimization that reduces person-level back-to-back conflicts:

- Builds a person-to-bib mapping to identify which people are in which heats
- **Normally excludes professionals** (they dance in too many heats to avoid conflicts)
- **When `allowDuplicateEntries` is on**, includes professionals so section events get spaced apart
- Iterates up to 50 passes, swapping adjacent heats when it reduces total conflicts without creating event adjacency

## 6. Heat Naming Conventions

### Event Names
- Standard: `"Pro-Am Syllabus Bronze Smooth Waltz"`
- Section: `"Pro-Am Syllabus Bronze Smooth Waltz - A"`, `"... - B"`

### Floor Heat Labels
- Displayed as letters: `"final (Heat A)"`, `"final (Heat B)"`
- Converted from `floorHeatIndex`: `String.fromCharCode(65 + index)`

### Combined Heat Labels
- When multiple events share a heat: `"Bronze Waltz + Silver Waltz"`
